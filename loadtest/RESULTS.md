# CP5 負荷テスト結果 — 売り越しゼロのままスケール

## 構成
- DB: Amazon Aurora DSQL マルチリージョン（東京 ap-northeast-1 ＋ ソウル ap-northeast-2 ＋ witness 大阪 ap-northeast-3）
- 在庫: 100（単一商品・単一在庫行）
- 負荷: k6 v2.0.0、3000 リクエスト、50 VUs、東京/ソウルの2エンドポイントへ振り分け
- 購入処理: 在庫チェック＋減算を1トランザクション。OCC競合（SQLSTATE 40001 / OC000）は指数バックオフ＋ジッタでリトライ。

## 結果

| 指標 | リトライ8回 | リトライ40回 |
|---|---|---|
| confirmed（成功） | 100 | 100 |
| sold_out（売切れ409） | 2810 | 2900 |
| errors（500） | 90 | **0** |
| 最終在庫 | 0 | 0 |
| 売り越し | なし | なし |

- どちらも **confirmed = ちょうど100 = 在庫数**、最終在庫 0。**売り越しは一度も発生しない**。
- リトライ8回では、極端な競合下で90件が OCC 競合(40001)を解消しきれず 500 になった。これは売り越しではなく「競合で諦めた購入」（在庫は減っていない）。
- **リトライ予算を40回に増やすと競合を全件さばき切り、エラー0**。3000 リクエスト中、200(成功100)/409(売切れ2900)のみで、サーバーエラーゼロ。

## 結論
リクエスト量に関係なく、販売数が在庫数を超えることはない。DSQL のマルチリージョン強整合＋アプリ側の OCC リトライにより、世界中から同時に殺到しても「ちょうど在庫数だけ売れて、それ以上は売れない」を実機で示せた。

## 再現手順
```bash
MR_STOCK=100 npm run db:mr-setup      # 在庫100にリセット
npm run dev                            # 開発サーバー
VUS=50 ITER=3000 k6 run loadtest/drop.js
```
